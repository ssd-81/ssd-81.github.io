## TL; DR
- I did not win. 
- My project was not deployed; the main product loop was too simple. Execution was sloppy. 
- Demo failed. Finally tasted the "final day demo failure" experience. 

## Done
This was a long process; and there was a lot of feedback I got on the process. 
A few points I would like to keep in mind: 
1. Build keeping in mind the user; try to make it useable regardless if the product is clunky or too complex. A useble product has far more value than one with ideas that are on just paper. Execute the loop properly. 
2. Execute one level deeper; execution beats ideation 99 % of the times (I make up figures). Tbh, building reveals a lot more than simply thinking about the process will. Try to make the product touch ground. 
3. Iteratively introduce complexity and with reasoning. Producing an artifact that is too complex from the start will not get the work done. Challenge your own decisions. 

## Where I lacked? 
If I think and read across the context; the major missing point was execution and iteration on feedback. I should have built something faster and likely tried to get feedback on that. A great way to actually build something great is to first build something and then showcase it for feedback and keep building from there. 

I took too long in the idea domain and produced an artifact too late. Producing something faster and then taking it up from there is the name of the game. If only I take up this single learning and compound with this idea, it is very valuable in itself. Try to shorten the loop from idea to execution and ask real people for feedback rather than waiting from final judgement day. 

Communication is going to evolve rapidly from here. With the advent of fast rapid prototyping, the idea that communication is a soft skill is not going to last very long. It is eventually going to become a backbone. 
- It is fine to not have the perfect style or cadence, but asking questions and getting feedback is going to become ever more important. 
- Building in isolation is going to become a problem. Feedback is going to become a core part of the loop. 

## What from here? 
The experience did leave me with a lot of ideas and questions. I did realize the value of a broken product which can actually be used by a user to pain points that are faced by users to product thinking and how to bridge gaps between compliance and user experience. 

The core idea that I want to take up from this experience is pretty simple: build before you speak. I am going to focus on understanding the problem from a uninformed lens and then use the product as a vehicle to build deeper understanding of the problem and the user.
